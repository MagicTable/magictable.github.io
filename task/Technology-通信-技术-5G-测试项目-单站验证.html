Technology - 通信 - 技术 - 5G - 测试项目 - 单站验证

魔术桌2025年3月17日任务总结/技术进度-待完善知识总结

文章摘要： 文章摘要。

介绍

作用

  1. 确保基站设备状态。（确保基站是开通状态）
  2. 核查基站性能指标。（确保性能达标）
  3. 核查基站内小区是否接反。
  4. 核查基站内外切换问题。（小区与小区之间的切换）
  5. 核查基站覆盖情况。

所需物品

  • 携带介绍信、个人身份证、工卡。
  • 携带测试手机、GPS、加密狗、个人电脑。
  • 携带逆变器。（确保电脑电量充足）
  • （建议）携带移动电源、充电线、U盘、扩展坞、水杯、雨伞、应急食物、现金。

关注指标

  • RSRP、SINR、PCI、上传下载速率、频点、Ping时延等

流程和注意事项

前 - 测试准备工作

测试项

  • 获取现网数据：网优工程师从无线工程师获取现网配置数据。
  • 核查数据是否一致：检查实际数据和规划数据是否一致。

注意事项

  • 请认真查看下达任务时提供的“工参”文件。
  • 寻找到正确地点的正确基站和小区。
  • 环绕一圈，首次观察小区PCI是否出现基站馈线接反。

前 - 基站设备检查

测试项

  • 基站安装检查
  • 基站故障检查
  • 基站告警检查

注意事项

  • 通知后台查看当前是否有影响业务的报警。（确保不存在影响业务的告警）
  • 上站，确保上站效率，并通知后台检查5G各参数配置、SCTP链路等。
  • NSA组网下需确认LTE和5G小区是否存在告警以及问题是否解决。
  • LTE和5G小区的X2口是否正常建立。

前 - 基站参数检查

测试项

  • 检查“频点、TAC、PCI、PRACH、经度、纬度、方位角、下倾角，天线挂高”等参数与规划数据一致，确保邻区完整。

注意事项

  • 现场核查站点经纬度，小区方位角，下倾角，挂高等信息，如规划有误，记录并更新。

中 - 现场照片收集

测试项

  • 天面照片、配电信息、有无空余天支等

注意事项

  • 照片应清晰，能分辨出天线安装位置，尤其是是否有空余抱杆信息、能分辨出楼顶抱杆、楼顶增高架、铁塔、管塔、美化树等信息。

中 - 定点性能验证（CQT测试）

测试项

  • Ping时延、FTP上传测试、FTP下载测试、附着（入网测试）/去附着（退网测试）/（attach\dettach）（模拟飞行模式）、语音业务、切换测试

注意事项

  • 验证测试结果是否达标。
  • 一种类型测试，按小区记录log，每个站点（按3个小区计算）共计21个log，log应命名规范统一。

中 - 路测覆盖验证（DT测试）

测试项

  • 进行站点的绕圈覆盖路测，验证该站点三扇区覆盖范围是否合理，是否存在天馈接反的情况，是否存在越区覆盖，切换功能是否正常。

注意事项

  • 不但要验证同站3小区之间的切换是否正常，还应验证每个小区对周围站点的切换是否正常，共计1个log。

中 - 问题报告分析

测试项

  • 路测过程中，如站点小区存在接反情况，需联系施工队进行处理；如小区存在覆盖过远或覆盖不足情况，需通过RF调整进行覆盖优化；如存在切换问题，需核查4G/5G邻区是否完整，如有缺失邻区，及时添加，并进行复测。
  • CQT测试时，需注意各项测试指标是否达标，如不达标需要进行定位分析，提出解决方案，并及时处理，处理后，及时进行复测。
  • 在RF优化调整过程中，如存在优化调整受限的小区（美化天线、安装不规范等）情况，需联系施工队进行整改。

注意事项

后 - 数据收集整理

测试项

  • 收集相关log，共计22个，天面照片1张。
  • 对提交的照片、log命名、数量、遗漏测试项做好登记，方便后期补测，对补测数据及时做好更新。

注意事项

  • 数据整理命名规范。
  • 站点单验完成，相关数据应当天上交专人进行收集整理。
  • 做好遗留问题记录，方便二次上站补测，建议以邮件形式发送，方便日后备查。
  • 对单验通过提交的log及时做好整理及记录工作，尤其做好缺失log登记，方便补测。
  • 主要单站log包中，只保留达标的log，不达标的log应及时删除，即每个站点总log数不能超过22个。

后 - 输出单验报告

测试项

  • 按照单验新模板要求整理“单站验证报告”，发现测试未达标项，做好记录并及时补测。

注意事项

  • 根据报告模板规范。
  • 报告所有必填项应填报真实、准确，不得遗漏，如有公式，需去掉公式。
  • 报告名称应命名统一，且统一按照新模板输出。
  • 补测数据及时更新，确保报告内容与log一致，1个站点只能维护1个报告。
  • 补测人员上站前应从负责人处获取该站点所有问题，做到一次上站所有问题全部清除，避免2次上站。

测试业务和指标

注

测试业务的达标指标根据具体项目要求来定。

CQT测试 - Ping时延（Ping）

测试要求

  • 32Bytes小包：ping 100次，间隔2秒，时延平均小于15ms，成功率大于99%。
  • 2000Bytes大包：ping 100次，间隔2秒，时延平均小于17ms，成功率大于99%。

LOG要求

  • 每个小区打包和小包各1个。（共6个）

注意事项

  • 小包时延多在15~20ms之间。
  • 大包时延多在20~30ms之间。
  • 仅统计RAN侧时延，需扣除传输链路和核心网侧时延。

CQT测试 - 下载吞吐量（FTP Download）

测试要求

  • 好点：-80dBm <= SS-RSRP < -70dBm、15dB <= SS-SINR < 20dB、64/32通道 > 800Mbps、8通道 >400Mbps、4通道 > 260Mbps
  • 差点：-100dBm <= SS-RSRP < -90dBm、-5dB <= SS-SINR < 0dB、64/32通道 > 90Mbps、8通道 >45Mbps、4通道 > 30Mbps

LOG要求

  • 每个小区1个。（共3个）

注意事项

  • 速率稳定后开始记录LOG，保持30s以上。

CQT测试 - 上传吞吐量（FTP Upload）

测试要求

  • 好点：-80dBm <= SS-RSRP < -70dBm、15dB <= SS-SINR < 20dB、64/32通道 > 120Mbps、8通道 >60Mbps、4通道 > 50Mbps
  • 差点：-100dBm <= SS-RSRP < -90dBm、-5dB <= SS-SINR < 0dB、64/32/8通道 > 1Mbps、4通道 > 0.8Mbps

LOG要求

  • 每个小区1个。（共3个）

注意事项

  • 速率稳定后开始记录LOG，保持30s以上。

CQT测试 - 语音（CSFB）

测试要求

  • EPS FB（主叫）：成功率100%，呼叫建立时延小于5秒。
  • EPS FR（被叫）：成功率100%，返回时延小于2.5秒。

LOG要求

  • 每个小区1个。（共3个）

注意事项

DT覆盖（环测/路测）

待完善

测试要求

  • 采集站点小区的PCI,RSRP,SINR、上传速率，下载速率等。
  • 验证基站上的小区参数是否按规划要求设置。
  • 测试小区是否有天馈线接反现象。
  • 测试小区与周边小区是否正常切换。

LOG要求

  • 共2个。（ping 2000byte和ping 32byte）

注意事项
