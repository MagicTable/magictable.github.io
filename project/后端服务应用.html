项目规范文档

缩进

所有前端项目全局采用4个空格缩进。

<div>
    <button></button>
</div>

命名

  • 变量名、函数名、模板标签的ref、props属性：蛇形命名法（user_name）
  • 所有的组件文件名称（包括通用组件）、业务组件名称、保存了组件的变量、业务的组件标签、类型：大驼峰命名法（UserName）
  • 第三方库函数名：维持原库命名，一般为小驼峰命名法（userName）
  • hook函数：小驼峰命名法（userName）
  • 通用组件标签、html样式类名、组件插槽命名、组件标签上的组件：短横杠命名（user-name）

命名前缀

  • 所有api封装的函数以api_开头。
  • 所有全局状态实例以store_开头。
  • 所有的数据转换函数以transform_开头。

组件编写

组件根标签的顺序始终应该是：script脚本、template结构、style样式、

<script setup>
</script>

<template>
</template>

<style scoped>
</style>

组件属性应尽可能都有默认值，即实现在没有传入参数时也能正常使用组件。

一个样式的的大多数组件状态应该使用reactive定义在一起，并命名为state，尽量避免使用ref，所有的方法都使用一个对象包裹，对象命名为methods

vue状态管理和方法管理规范

组件内部状态管理

  • 组件内部的状态统一放在一个reactuve里面，并命名给state。
import { reactive } from "vue"
const state = reactive({
    // 这里存放所有的状态
});

编程思想

统一使用export进行导出模块内容，禁止使用export default和零散的export const。

// good
export function myFunction() {
  // ...
}
export class MyClass {
  // ...
}

// good
const a = 1;
const b = 2;
// some code
export { a, b }; // 集中导出

避免嵌套除非无法避免。

只做有用的封装，反复思考封装之后与封装之前的改变。

时刻考虑侵入性、可热插拔，但具体应用需要灵活取舍。

Web层的axios请求服务命名规范

请求路径的变量命名：通过下划线拼接多个层级的路径，并全小写，例如：/user/logon、user_logon
请求的请求体中的数据命名规范：使用后端的小驼峰命名法，即在发生和接收时进行“前端的下划线命名”和“小驼峰命名”之间的转换。

Controller控制层请求URL映射规范

请求数据的接收：接收小驼峰命名法的对象参数
请求RUL路径映射：全小写
